System Architecture

üèóÔ∏è Dual-Database Strategy

The project employs a sophisticated dual-database architecture to separate operational transactions from analytical processing, ensuring optimal performance for both real-time operations and historical analysis.

üíº Operational Database

Handles day-to-day transactions and stores the latest 30 days of data in a normalized relational schema.

  • ‚Üí Order information tracking
  • ‚Üí Customer data management
  • ‚Üí Product details storage
  • ‚Üí Staff interaction logs
  • ‚Üí Real-time shipping updates

üìä Analytical Database

Archives past 3 months of data in a star schema format optimized for complex analytical queries.

  • ‚Üí Central fact table (sales_fact_table)
  • ‚Üí Product dimension table
  • ‚Üí Date dimension table
  • ‚Üí Location dimension table
  • ‚Üí Fulfillment dimension table

üîÑ ETL Pipeline

Implemented using Python and SQL to seamlessly transfer data between operational and analytical databases.

1. Extract

Data is extracted from the operational database on a scheduled basis

2. Transform

Data quality checks, consistency validation, and schema transformation to star schema format

3. Load

Clean data is loaded into analytical fact and dimension tables for strategic analysis
